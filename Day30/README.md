# Day 30: FreeRTOS 記憶體管理 (Memory Management) 與配置策略

今天我們來到了 FreeRTOS 系統架構的最底層：**記憶體管理**。
探討了如何解決碎片化危機、五大 Heap 演算法的抉擇，以及航太級系統求穩的終極心法。

## 1. 致命絕症：記憶體碎片化 (Memory Fragmentation)
當系統頻繁地動態配置與釋放記憶體時，連續的空間會被切割成散落的碎塊。當有大陣列需要連續空間時，即使總剩餘記憶體足夠，`malloc` 依然會因為找不到「連續」車位而宣告失敗，導致系統崩潰。

## 2. 業界黃金標準 `heap_4` 與相鄰合併 (Coalescing)
FreeRTOS 的 `heap_4` 演算法完美解決了碎片化問題：
* **隱藏標頭**：每次配置記憶體時，底層會偷偷多切 8 Bytes 作為標頭，記錄這塊記憶體的大小。當呼叫 `free` 時，指標倒退就能查到大小。
* **相鄰合併 (Coalescing)**：在退房時，演算法會檢查該區塊的「物理前、後鄰居」是否也是空閒的。如果是，就會直接打通牆壁，將碎塊融合成一個連續的大空間，實現記憶體的自我修復。

## 3. FreeRTOS 五大 Heap 演算法抉擇
1. **heap_1 (鐵公雞)**：只能 `malloc` 不能 `free`。零碎片化，絕對穩定，專用於航太與醫療的極高安規系統。
2. **heap_2 (已被淘汰)**：具備 `free` 但不會合併，會產生嚴重碎片化。
3. **heap_3 (標準庫包裝)**：直接呼叫編譯器內建的 `malloc/free`，並加上中斷保護確保執行緒安全 (Thread-Safe)。
4. **heap_4 (業界預設)**：內建相鄰區塊合併神技，適用於 95% 的動態分配專案。
5. **heap_5 (空間魔術師)**：具備 `heap_4` 技能，並能透過 `vPortDefineHeapRegions()` API，將物理上分離的多塊實體 RAM (如內部 64KB SRAM + 外掛 8MB SDRAM) 縫合成一個統一調度的超級記憶體池！

## 4. 航太級安規：靜態分配 (Static Allocation)
在 ISO 26262 (汽車電子) 或高可靠度系統中，極度推崇**靜態分配** (`xTaskCreateStatic`) 而非動態分配。
* **錯誤發生在編譯期**：靜態分配是在寫程式碼時直接宣告全域陣列，若記憶體不足，會在編譯時直接報錯燒不進晶片。完全杜絕了在高速公路上運作到一半突然發生的 OOM (Out of Memory) 當機！
* **時間確定性 (Determinism)**：不需要花時間搜尋空餘的 Linked List，執行時間永遠是極速常數。
