# Day 28: FreeRTOS 資源守護者 (Mutex, Priority Inheritance & Semaphores)

今天我們解鎖了 RTOS 面試中最高頻率、最具技術深度的核心概念：**互斥鎖 (Mutex) 與優先權機制**。
我們透過火星探測車的經典 Bug 案例，徹底理解了作業系統如何保護資源與防禦死鎖。

## 1. 🚨 世紀 Bug：優先權反轉 (Priority Inversion)
* **災難情境**：低優先權任務 (Low) 拿著鎖，高優先權任務 (High) 等待鎖而休眠。此時不需要鎖的中優先權任務 (Mid) 醒來，搶占了 CPU (因為 Mid > Low)。結果 Low 永遠無法解鎖，導致 High 永遠被卡死。
* **致命後果**：最高優先權（最重要）的任務，實質上被中等優先權的任務給餓死了。

## 2. 🛡️ 終極防禦：優先權繼承 (Priority Inheritance)
FreeRTOS 的 `Mutex` 之所以比 `Semaphore` 適合保護記憶體，就是因為它內建了「優先權繼承」。
* **臨時提拔**：當 High 等待 Low 的鎖時，RTOS 老闆會把 Low 的優先權「臨時提升」到跟 High 一樣高。這讓 Mid 無法搶占 Low。
* **權力收回 (`Mutex_Give`)**：當 Low 歸還鎖時，老闆會瞬間把 Low 的優先權打回原形，並立刻喚醒門外的 High 順利接手資源。

## 3. 💀 巢狀呼叫的陷阱：自我死鎖 (Self-Deadlock)
* **當機原因**：任務拿到鎖後，如果不小心呼叫了內含「拿同一把鎖」的副函式，任務會因為鎖已被自己拿走而把自己敲暈休眠。由於只有自己能解鎖，導致永遠死結。
* **原廠解藥：遞迴互斥鎖 (Recursive Mutex)**。它具備「人臉辨識」與「計數」功能。發現是同一個任務來要鎖便會直接放行並將計數器 +1。借幾次就必須 `GiveRecursive` 幾次，計數器歸零才真正解鎖。

## 4. 🚗 資源池的自動車位計數器：計數信號量 (Counting Semaphore)
* **Mutex (0 或 1)**：適用於單一資源保護（如 UART 傳輸線、單一全域變數）。
* **Counting Semaphore (0 到 N)**：適用於「複數資源」管理（例如 2 顆硬體解碼器）。預設初始值為 N，前 N 個任務可以順利取得資源，第 N+1 個任務會自動被敲暈排隊，直到資源被釋放才自動喚醒，完美省去寫 `if-else` 的麻煩。
