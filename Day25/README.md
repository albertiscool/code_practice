# Day 25: FreeRTOS 多工作業系統核心解析

今天我們正式跨入了嵌入式系統的巔峰領域：**RTOS (即時作業系統)**。
從裸機的單一 `while(1)` 思維，升級到了時間切片、上下文切換、以及精細的記憶體監控！

## 1. 🔄 上下文切換 (Context Switch) 與專屬背包
在 RTOS 中，CPU 是如何做到「同時」處理多個任務的？
* **硬體基礎**：依賴 `SysTick` 定時器（通常 1ms 觸發一次中斷）。
* **專屬 Stack**：每個被創建出來的任務，都有自己的身分證 (`TCB`) 和獨立的記憶體背包 (`Stack`)。
* **切換三部曲**：
  1. **Save**：把當前 CPU 暫存器做到一半的資料，全數塞進當前任務的 Stack 裡保存。
  2. **Schedule**：RTOS 老闆更新指標 `pxCurrentTCB`，決定換下一個最高優先權的任務上場。
  3. **Restore**：把新任務 Stack 裡的資料彈出，還原回 CPU 暫存器中繼續執行。
* **代價 (Overhead)**：頻繁切換會吃掉 CPU 約 0.4% 的算力，這被稱為「上下文切換開銷」。雖然有負擔，卻換來了致命錯誤隔離與緊急任務插隊 (Preemption) 的無價優勢！

## 2. 🛑 狀態切換盲點：絕對禁止死等 (Busy Waiting)
* **裸機毒藥**：如果在高優先權任務裡寫了 `for(int i=0; i<100000; i++);`，該任務會持續霸佔 `Running` 態，導致低優先權任務永遠拿不到 CPU 資源，慘遭「餓死 (Starvation)」。
* **RTOS 解藥**：必須呼叫原廠 API 如 `vTaskDelay()`。這會讓任務主動切換到 **`Blocked` (阻塞休眠態)**，告訴排程器：「我這段時間不需要 CPU，請分給別人用。」，實現 100% 的系統利用率。

## 3. 💥 Stack Overflow 當機與防禦
* **災難爆發**：如果在任務內宣告了過大的陣列，資料會撐破該任務專屬的 Stack 邊界，踩爛別人的記憶體，觸發 ARM 核心的硬體死機 (Hard Fault)。
* **防呆機制**：開啟 `configCHECK_FOR_STACK_OVERFLOW`，讓 RTOS 在切換任務時順便檢查包包有沒有破洞，若有則觸發 Hook 函式點亮紅燈報警。

## 4. 📏 神級優化：水位線監測 (High Water Mark)
要如何知道給任務的 RAM 是給太多還是給太少？
* **藍色墨水原理**：RTOS 在開任務時，會把 Stack 空間全數填滿特殊圖騰（如 `0xA5A5A5A5`）。
* **極限偵測**：程式運作中，呼叫 `uxTaskGetStackHighWaterMark()`，OS 會從背包最底部開始計算「還剩下多少未被破壞的藍色墨水」。
* **功用**：這個數值代表「歷史上最少還剩下多少安全空間」。工程師能藉此精準縮減過大的 Stack 空間，極致榨乾微處理器每一滴珍貴的 RAM 資源！
