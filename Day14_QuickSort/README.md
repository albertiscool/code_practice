# Day 14: 排序演算法的霸主 - 快速排序法 (Quick Sort)

## 1. Quick Sort 的核心魔法：保證鎖死
Quick Sort 是實戰中平均速度最快 ($O(N \log N)$) 的排序法，C 語言的 `qsort()` 底層便是基於此演算法。
* **分家 (Partition)**：每次抓出一個基準點 (Pivot)，將比它小的丟左邊，大的丟右邊。
* **保證鎖死**：洗牌完的瞬間，Pivot 會精準降落到它這輩子「絕對正確的最終位置」，永不移動。
* **信任小弟 (遞迴)**：將 Pivot 左右兩邊的爛攤子交給遞迴處理，直到每個數字都被鎖死。

## 2. 指標 Swap 的觀念釐清
呼叫 `swap(&arr[i], &arr[j])` 時，並不是把記憶體的實體板塊對調！記憶體位址 (房間號碼) 是永遠不動的，我們是利用指標拿到鑰匙，打開房門把裡面的「值 (住客)」互換。

## 3. 致命退化與「三數取中法」
* **最壞情況 (Worst Case)**：若固定挑選最後一個元素當 Pivot，且陣列剛好是「已排序」或「反向排序」，陣列將無法對半切（變成 0 與 N-1），時間複雜度慘跌至龜速的 **$O(N^2)$**。
* **業界解法 (Median-of-Three)**：取出陣列的頭、尾、正中間三個數字，取這三者的「中間值」作為 Pivot。這能完美避開極端情況，確保陣列被漂亮地對半切開。

## 4. 韌體工程師的夢魘：Stack Overflow
在 MCU 微控制器中，Stack (堆疊區) 容量極小。當 Quick Sort 退化成最壞情況時，深度達 N 層的遞迴會瞬間塞爆 Stack，引發硬體錯誤 (HardFault) 導致系統死機。這也是高安全規範 (如車規 MISRA-C) 嚴格禁止使用遞迴的主因。

## 5. 面試耍帥題：純位元運算的 XOR Swap
不使用 `temp` 變數完成交換的神級寫法：
```c
void swap(int *a, int *b) {
    *a = *a ^ *b;
    *b = *a ^ *b;
    *a = *a ^ *b;
}
```
* **原理**：利用 XOR 的「自我抵銷 (`A^A=0`)」特性在半空中完成交換。
* **實戰反思**：若 `a` 與 `b` 指向同一位址，資料會瞬間歸零蒸發！且現代編譯器優化傳統 `temp` 寫法的速度反而更快，故實戰中應避免使用此花招。
