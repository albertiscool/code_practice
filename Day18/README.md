# Day 18: 硬體中斷 (Interrupt) 與前背景架構 (ISR)

今天我們進入了微處理器 (MCU) 最核心的運作機制：「中斷 (Interrupt)」。這是在系統廠或 IC 設計廠面試韌體工程師時，絕對必考的「鑑別度大題」。

## 1. ⚡ 前背景架構 (Foreground/Background Architecture)
在 MCU 開發中，系統通常分為兩層：
* **背景 (Background)**：`main` 函式裡面的 `while(1)` 無窮迴圈。負責處理低優先級、耗時的重度任務（如更新螢幕、處理封包）。
* **前景 (Foreground)**：中斷服務常式 (ISR)。由硬體強制觸發，擁有最高優先權。

就像看書看到一半去接電話一樣，CPU 會放下 `main` 的工作，去執行 ISR，執行完再回來繼續。

## 2. 🚨 ISR 的三大鐵律 (面試必考)

### 鐵律一：快、狠、準
* **絕對禁止**在 ISR 裡面執行耗時操作（如 `printf`、`delay` 或漫長的迴圈）。
* **最佳實踐**：在 ISR 裡面只做一件事：「把一個全域變數 (Flag) 設為 `true`」，然後立刻離開。把真正的耗時工作交回給 `main` 迴圈去「延遲處理 (Deferred Processing)」。

### 鐵律二：跨界溝通的橋樑 `volatile`
如果全域變數 `g_button_pressed_flag` 沒有加上 `volatile`，在編譯器開啟 `-O3` 最高優化時，會引發**「死亡優化陷阱」**。
* `main` 迴圈會自作聰明把變數快取 (Cache) 到 CPU 內部的 Register 裡。
* 當 ISR 將實體記憶體的數值改為 `true` 時，`main` 迴圈永遠只會看著自己手上的 `false` 影印本，導致系統對按鈕毫無反應。
* `volatile` 強制 CPU 每次都去實體記憶體讀取最新狀態！

### 鐵律三：必須手動清除旗標 (Clear Pending Flag)
* 進入 ISR 後，必須立刻操作硬體暫存器（如 `EXTI->PR |= (1<<0);`）來清除「中斷掛起旗標」。
* **如果不清的後果**：CPU 只要一回到 `main` 函式，硬體就會立刻舉手說「又有人按鈕了」，導致 CPU 在一微秒內再次跳進 ISR。這會引發 **「中斷風暴 (Interrupt Storm)」**，讓主迴圈徹底餓死，整台機器死當崩潰。

## 3. 🛡️ 軟體測試術語：Mocking (打樁/模擬)
因為我們在個人電腦 (Windows/x86) 上沒有真正的微處理器硬體，我們會在程式碼中使用 `[Mocked]` 的標籤。
這代表我們用軟體邏輯去「假裝」硬體發出了訊號，藉此在沒拿到實體電路板之前，就能先在電腦上測試 ISR 與 `main` 迴圈的互動邏輯。

## 4. 🪲 巨集定義的語法除錯 (Macro Syntax)
在進行結構體映射時，必須嚴格遵守 C 語言的強制轉型語法：
* **基底位址**：應該是純數字，如 `#define EXTI_BASE 0x40013C00U`
* **指標轉型**：必須把 `*` 包在型別括號內，如 `#define EXTI ((EXTI_TypeDef *)EXTI_BASE)`。
* ❌ 絕對不能寫成 `(*(volatile uint32_t)...)`，這會變成試圖去「提領」一個尚未綁定結構體的裸數字，導致編譯失敗。
