# Day 27: FreeRTOS 任務通訊 (Queue)、同步機制與記憶體陷阱

今天我們跨越了單純的任務排程，進入了多工系統最棘手的領域：**資料共享與同步 (Synchronization)**。
我們用生活化的比喻，徹底拆解了多執行緒開發中最惡名昭彰的死機雷區！

## 1. 💣 全域變數的悲劇：Race Condition (競態條件)
在搶占式多工環境下，絕對禁止使用全域變數傳遞資料！
* 當多個任務試圖同時修改同一個變數時，因為 Context Switch (中斷切換) 隨時會發生，任務 A 修改到一半的資料，會被任務 B 粗暴覆蓋，這稱為 **Race Condition**。
* **解藥：原子操作 (Atomic Operation)**。FreeRTOS 的 Queue 在底層會使用 `taskENTER_CRITICAL()` 關閉所有硬體中斷，把讀寫動作強力黏死成「不可分割」的一瞬間，保證資料的絕對安全。

## 2. 🌪️ 多核心的極致防禦：自旋鎖 (Spinlock)
在多核心 CPU 中，單純關閉中斷無法阻止「隔壁核心」同時搶奪記憶體。
這時必須動用硬體級的 **Spinlock (自旋鎖)**。當 Core 1 佔用資源時，Core 2 不會去休眠，而是在原地跑死迴圈 (`while`) 狂敲門。雖然浪費一點點 CPU 資源，但在微秒級的保護下，換來的是零 Context Switch 開銷的極速反應！

## 3. 🍔 完美餐廳架構：解耦 (Decoupling) 與 Queue 連動
FreeRTOS 的 Queue 就像是廚房與外場之間的「出餐吧台」，徹底解開了硬體與演算法之間的生死綁定。
* **自動阻塞 (Blocking)**：Queue 空了去讀、或是滿了去寫，任務都會被自動敲暈進入 `BLOCKED` (休眠態)，不浪費一滴 CPU 算力。
* **神級連動喚醒**：消費者從滿水位拿走一筆資料，Queue 會自動喚醒在睡覺的生產者；反之亦然。這達成了完全自動化的流量控制。

## 4. 🚨 絕對禁區：ISR 專用 API (`FromISR`)
硬體中斷 (ISR) 凌駕於作業系統之上，**絕對不允許休眠 (Block)！**
如果在 ISR 內呼叫普通的 `xQueueSend` 試圖等待，系統會立刻崩潰 (Hard Fault)。在 ISR 內，必須強制使用字尾帶有 `FromISR` 的特製 API，它遇到 Queue 滿時會瞬間回傳失敗並離開，確保中斷在幾奈秒內結束。

## 5. 💀 傳遞大資料的雙重死亡陷阱
為了效能，傳遞 1MB 的影像資料絕對不能 Copy by Value，必須**「傳遞指標 (Pointer)」**。但這會帶來記憶體生命週期的兩大難題：
* **記憶體洩漏 (Memory Leak)**：生產者用 `malloc` 借用 Heap 空間，擁有權透過 Queue 轉移後，消費者處理完卻忘記呼叫 `free` 歸還，導致系統記憶體耗盡死機。
* **懸空指標 (Dangling Pointer)**：生產者偷懶，將「區域變數 (Local Variable，存在 Stack 裡)」的指標傳入 Queue。當函式結束時，記憶體被系統強制退房回收。消費者拿到指標時，該空間已被其他中斷的垃圾資料覆蓋，導致靈異的亂碼 Bug。
