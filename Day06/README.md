# Day 6: static、volatile 與巨集陷阱大解密

## 1. 百變怪 `static` 的兩種分身
在 C 語言中，`static` 放在不同地方會有截然不同的效果：
* **區域靜態變數 (Local Static Variable)**：放在函式內。它不會存在堆疊 (Stack) 而是存於 Data/BSS 區。它只會初始化一次，且函式結束後它的值**不會消失**，實戰中常用作計數器或狀態機。
* **全域靜態變數 (Global Static Variable)**：放在檔案最外層。它會將變數的「可見度」**鎖死在當前的檔案內**，別的檔案絕對無法透過 `extern` 偷拿去用，適合用來隱藏底層參數。

## 2. 韌體保命符：`volatile`
`volatile` 用來警告編譯器：「這塊記憶體隨時會被外部力量改變，**你不准對它做任何最佳化，給我老老實實地去真實記憶體 (RAM) 讀寫！**」
* **情境 A (硬體暫存器/Input)**：常搭配 `const volatile` 宣告。`const` 防止軟體白目去改寫輸入腳位，`volatile` 確保每次讀取都是硬體當下的最新電壓，防止編譯器誤以為值沒變而把程式碼刪掉。
* **情境 B (多執行緒/ISR 溝通)**：若不用 `volatile`，編譯器為了加速，會把變數「快取 (Cache)」在 CPU 暫存器中。當硬體中斷 (ISR) 改變了 RAM 裡的值，主程式依然會死盯著暫存器裡的舊資料，導致 `while` 等待迴圈永遠卡死！

## 3. 系統廠神級巨集：硬體暫存器操作
在嵌入式系統中，我們不能宣告指標變數來操控硬體，因為會浪費極其珍貴的 RAM。業界標準做法是使用 `#define` 搭配強制轉型與提領：
```c
#define GPIO_CTRL  (*((volatile uint32_t *)0x40021000))
```
* **解讀**：把 `0x40021000` 強制轉型成「不可被優化的 32位元指標」，然後在最外面加上 `*` 直接提領出那塊空間。
* **好處**：完全不浪費 RAM 空間，且能像普通變數一樣直接讀寫硬體：`GPIO_CTRL = 1;`。

## 4. 巨集本質與致命陷阱
* **本質**：`#define` 只不過是預處理器在編譯前執行的**「高級純文字複製貼上」**。
* **函式巨集陷阱**：因為只是文字替換，如果寫作 `#define SQUARE(x) x * x`，當傳入 `SQUARE(2+3)` 時，會被替換成 `2+3*2+3`，因為「先乘除後加減」而算出錯誤的 11。
* **保命鐵則**：在撰寫函式巨集時，**必須用括號 `()` 將所有參數和整體運算式嚴密包覆**！
  👉 正確寫法：`#define SQUARE(x) ((x) * (x))`
