# Day 2: 指標進階觀念與陷阱大解密

## 1. 指標運算 (Pointer Arithmetic)
當我們對指標進行加減法（例如 `ptr++` 或 `ptr + 2`）時，它**不是單純加上 Byte 數**，而是**「加上資料型態的大小」**。
* 公式：`新位址 = 原位址 + (移動步數 × 資料型態的 Byte 數)`
* 範例：若 `int *p` 位於 `0x1000`，`p + 2` 會走到 `0x1000 + (2 × 4 Bytes) = 0x1008`。
* **注意**：陣列名稱（如 `arr`）本質上是一個**常數指標**，位址是寫死的，不能被修改，所以 `arr++` 會引發編譯錯誤。

## 2. sizeof：陣列 vs 指標
許多新手常誤以為指標和陣列完全一樣，但用 `sizeof` 就能看穿它們的真面目：
* **`sizeof(arr)`**：會算出**整個陣列的總大小**。例如 `int arr[3]` 就是 `3 × 4 = 12 Bytes`。
* **`sizeof(ptr)`**：永遠只會算出**指標變數本身的大小**。
  * **什麼是 64 位元？** 現代 64 位元系統中，CPU 處理寬度與記憶體門牌號碼長度皆為 64 bits。為了裝得下這麼長的位址，指標的大小固定為 **8 Bytes**（`64 bits ÷ 8`）。

## 3. 常數指標大魔王 (const 在哪裡大不同)
面試必考題，判斷秘訣是「從右邊往左邊念」：
1. **Pointer to Constant (指向常數的指標)**：`const int *p`
   * 解讀：指標指向的內容是唯讀的。
   * 特性：可以叫指標去指別人 (`p = &b` ✅)，但不能透過指標改值 (`*p = 100` ❌)。
2. **Constant Pointer (常數指標)**：`int * const p`
   * 解讀：指標本身是常數，死心塌地。
   * 特性：位址綁死了不能指別人 (`p = &b` ❌)，但可以透過它改值 (`*p = 100` ✅)。

## 4. `arr` 與 `&arr` 的致命差異
雖然 `printf("%p", arr)` 和 `printf("%p", &arr)` 印出來的記憶體位址一模一樣（都是陣列的起點），但它們的**型態與眼界**完全不同：
* **`arr` (房間管理員)**：退化為指向「第一個元素」的指標。
  * `arr + 1` 會往前跨一個元素的距離（例如 2 Bytes 的 short）。
* **`&arr` (大樓管理員)**：指向「一整個陣列」的指標。
  * `&arr + 1` 會直接跨過**一整座陣列**的總長度（例如 `4 × 2 = 8 Bytes`），非常容易導致越界存取 (Out of bounds) 的致命 Bug！

## 5. `arr + 1` 與 `arr++` 的核心差異
很多新手會疑惑，為什麼 `arr + 1` 合法，但 `arr++` 卻會報錯？關鍵在於**「純運算」**和**「賦值修改」**的差別：
* **`arr + 1` (合法)**：這只是一個單純的「數學運算式」，就像算 `5 + 1 = 6`。電腦只是算出一個新的記憶體位址數字，但陣列 `arr` 的本體依然穩穩地綁定在原點，完全沒有被改變。
* **`arr++` (報錯)**：它是 `arr = arr + 1` 的縮寫，這是一個**「賦值動作」**。因為陣列名稱 `arr` 在 C/C++ 中是一個常數指標 (Constant Pointer)，它出生時的位址就被寫死了，你不能強制修改它指向別人，所以編譯器會直接報錯阻擋。
* **解法**：如果真的想用 `++` 往前走，必須先宣告一個普通的代理指標（例如 `int *ptr = arr;`），然後執行 `ptr++` 才是合法的。
