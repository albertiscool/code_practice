# Day 19: UART 通訊協定與實體層波形 (Physical Layer)

今天我們進入了微控制器與外界溝通最重要的橋樑：「UART 通訊介面」。我們不僅從軟體層次探討了 CPU 如何與硬體互動，更深入到了實體電線上的電壓波形，這些都是韌體工程師面試的超級必考題！

## 1. 🐌 軟體層次的通訊：輪詢 (Polling) 機制
在最基礎的 UART 傳輸中，CPU 必須透過不斷檢查「狀態暫存器 (SR)」的旗標，來決定能不能動作：
* **傳送 (TXE - Transmit Empty)**：CPU 必須死等 `while((SR & TXE) == 0)`。如果不等信箱空了就塞入新資料，會導致前一個字元被覆蓋消失 (Data Overrun)。
* **接收 (RXNE - Receive Not Empty)**：CPU 必須死等 `while((SR & RXNE) == 0)`，直到外部裝置真的送電壓進來。

**🚨 輪詢的致命缺點 (Blocking)**：
在等待硬體傳輸的過程中，CPU 會卡死在 `while` 迴圈內，無法更新螢幕、無法處理按鍵，這在業界被稱為「阻塞 (Blocking)」，是極度浪費 CPU 效能的作法。

## 2. 🌊 硬體層次的通訊：UART 實體波形圖
當我們要傳送字母 `'A'` (`0x41` = `0100 0001`) 時，實體電線上的電壓變化如下：

```text
高電位 (1) -- Idle --+     +-- D0 --+                       +-- D6 --+          +-- Stop -- Idle --
                     |     |        |                       |        |          |
低電位 (0)           +-Start-+        +- D1-D2-D3-D4-D5 -+        +- D7 -+
```
* **Idle (1)**：平常保持高電位，一旦變成 0 就能瞬間抓出「電線斷掉 (Break)」的異常。
* **Start Bit (0)**：強制拉低製造「下降沿 (Falling Edge)」，喚醒接收端開始計時收信。
* **Data Bits (LSB First)**：從最低位元開始反著傳送。
* **Stop Bit (1)**：強制拉高回 1，確保下一個 Start Bit 絕對能製造出下降沿！

## 3. 🎯 訊號採樣的物理極限 (Oversampling)
為什麼接收端一定要抓每個 Bit 的「正中間」？
1. 避開電壓爬坡/溜滑梯的邊緣地帶 (Rise/Fall Time)，只抓最乾淨的穩定訊號。
2. 容忍「時鐘偏移 (Clock Drift)」。就算傳送方和接收方的石英震盪器速度有微小誤差，只要抓正中間，就有前後各 50% 的完美緩衝空間。

## 4. 漏洞百出的 Parity Bit 與工業標準
UART 內建的同位元檢查 (Parity) 只能抓出「奇數個」錯誤。如果被閃電打到導致 2 個 Bits 同時翻轉，Parity 算出來的偶數依然吻合，會讓系統吃下錯誤資料。
* **進階解法**：在現代網路、USB 或車載通訊 (CAN Bus) 中，都改用 **CRC (循環冗餘校驗)**，利用複雜的多項式除法，能抓出 99.999% 的連續錯誤。高階伺服器甚至會使用 **ECC** 來進行物理位元的瞬間修復。

## 5. 🚀 晶片架構的終極演進：Polling vs Interrupt vs DMA
面對大量的 UART 資料，現代晶片有三種處理境界：
1. **Polling (輪詢)**：CEO (CPU) 自己盯著一樓信箱死等。(系統癱瘓)
2. **Interrupt (中斷)**：信箱滿了打電話叫 CEO 下樓拿。(CEO 還是會因為頻繁拿信而累死)
3. **DMA (Direct Memory Access, 直接記憶體存取)**：CEO 聘請專職外送員 (DMA 控制器)。硬體收到信後，**DMA 會自動把字元搬移到 RAM 陣列中，完全不需要 CPU 介入**。等 100 封信全搬完了才通知 CPU 處理，這才是現代 MCU 達到效能巔峰的終極魔法！
