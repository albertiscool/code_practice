# Day 5: 結構體記憶體對齊 (Memory Alignment) 與底層封包實戰

## 1. 記憶體對齊與 Padding (空白填充)
為了讓 CPU 讀取記憶體的效能最大化，C/C++ 編譯器會自動進行「記憶體對齊」。如果變數沒有對齊在它規定的倍數位址上，編譯器就會自動塞入空白的 **Padding (填充位元組)**。
* **致命的排列順序**：如果把小變數和大變數交錯宣告 (例如 `char`, `int`, `char`)，中間會產生大量的 Padding，浪費將近 50% 的記憶體。
* **最佳化黃金法則**：宣告 Struct 時，**「把佔用空間最大的變數排在最上面，最小的變數排在最下面」**，就能讓變數緊密擠在一起，榨出最大空間！

## 2. 結構體的「總大小」陷阱
計算 Struct 大小時，除了中間的 Padding，絕對不能忘記「結尾的 Padding」：
* 規則：**整個 Struct 的總大小，必須是裡面「對齊要求最大的成員」的倍數。**
* 範例：如果 Struct 裡面最大的是 `double` (8 Bytes)，那總大小如果算出來是 18 Bytes，編譯器會在最後面強制補上 6 Bytes 的結尾 Padding，讓總大小變成 24 Bytes。

## 3. 基本資料結構大小複習
* **`char`**: 1 Byte
* **`short`**: 2 Bytes
* **`int`, `float`**: 4 Bytes
* **`double`, `long long`**: 8 Bytes
* **指標 (Pointer)**: 
  * 32 位元系統：4 Bytes
  * 64 位元系統：8 Bytes (因為需要 64 bits 的長度來存放龐大的記憶體門牌號碼)。

## 4. 硬體通訊必備：強制緊密排列 (Packed)
在做網通封包傳輸或 I2C/SPI 底層通訊時，多餘的 Padding 會導致封包錯位解碼失敗。為了解決這個問題，我們會強制要求編譯器「不准對齊」：
* **GCC / Clang (Linux/Mac)**：在 Struct 後面加上 `__attribute__((packed))`。
* **MSVC (Windows)**：在 Struct 上下包夾 `#pragma pack(push, 1)` 與 `#pragma pack(pop)`。
* **代價**：強制緊密排列會犧牲 CPU 的讀取效能，因此平常應順其自然對齊，只有在需要傳輸實體封包時才使用 Packed。

## 5. 業界超狂優化：指標標記 (Pointer Tagging)
因為對齊規則的關係，如果一個變數要求 8 Bytes 對齊（例如 `double` 或 64-bit 指標），它在記憶體中的起始位址，必然是 8 的倍數。
* **數學特性**：8 的倍數轉換成二進位後，**最後 3 個 bits 永遠是 `000`**。
* **神級應用**：底層引擎 (如 JavaScript V8 或 iOS 核心) 會利用這多出來的 3 個 `000`，偷塞入型別旗標 (Flags) 來紀錄這是一筆字串還是數字。等到要真實讀取位址時，再用位元運算把尾巴強制歸零還原位址，達成極致的記憶體利用率！
