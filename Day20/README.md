# Day 20: I2C 通訊協定 (Open-Drain、時序與軟體模擬)

今天我們進入了嵌入式系統中最經典的「多主多從」通訊協定：**I2C (Inter-Integrated Circuit)**。
我們深入解析了它獨特的硬體電路設計，以及如何透過軟體去「純手工繪製 (Bit-Banging)」它的波形。

## 1. 🛑 為什麼 I2C 必須使用「開漏輸出 (Open-Drain)」？
在 UART 中我們使用推挽輸出 (Push-Pull)，晶片可以直接將電壓推到 `3.3V` 或拉到 `0V`。但因為 I2C 的同一條線上掛著多顆晶片，如果 A 晶片往上推、B 晶片往下拉，就會引發**短路燒毀**！
* **I2C 的天才解法**：晶片內部被剝奪了往上推的能力 (開漏)。晶片只能選擇「把線拉低到 0」或是「雙手放開」。
* **上拉電阻 (Pull-up Resistor)**：在電路上外掛電阻連到 VCC，當所有晶片都「雙手放開」時，電阻會自然將線路拉回高電位 `1`。
* **線交和 (Wired-AND)**：只要線上有一百顆晶片裡的一顆把線拉低，整條線就是 `0`。沒有人會互相傷害。

## 2. 🌊 I2C 的四大關鍵時序 (面試必考)
I2C 有一個鐵律：**當 SCL (時脈線) 為高電位 `1` 時，SDA (資料線) 絕對不准變動！**
(只能在 SCL 為 `0` 的安全期寫資料，在 SCL 為 `1` 的時候讀資料)。

原廠利用打破這個鐵律，創造了絕不混淆的開始與結束訊號：
1. **START Condition (起始)**：當 SCL 為 `1` 時，SDA 發生「下降沿 (High -> Low)」。(引起所有人注意)
2. **STOP Condition (結束)**：當 SCL 為 `1` 時，SDA 發生「上升沿 (Low -> High)」。(通訊解散)
3. **Data Validity (資料傳輸)**：遵守鐵律。SCL 為 0 時改變 SDA；SCL 為 1 時感測器讀取 SDA。而且 I2C 是 **MSB First (從最高位元開始傳)**。
4. **ACK / NACK (第九個時鐘的簽收)**：
   傳完 8 個 Bits 後，Master 必須**主動放開 SDA** (閉上嘴巴)。
   此時 Slave 如果有收到資料，會用力把 SDA 拉低到 `0` (ACK)。如果 Slave 沒拉，上拉電阻會讓線維持 `1` (NACK / 出錯)。

## 3. 🦈 硬體除錯地雷：RC 充電曲線與鯊魚鰭波形
上拉電阻不能隨便選！實體電線上有「寄生電容」。
當晶片放開 SDA 線時，上拉電阻必須對電容「充電」，電壓才會從 0V 慢慢爬回 3.3V。
* **如果電阻選太大 (如 100kΩ)**：充電水流太慢，波形的上升邊緣會變成圓緩的**「鯊魚鰭 (Shark Fin)」**曲線。在 400kHz 高速通訊下，電壓可能還沒爬到 1 的標準，就又被拉低了，導致通訊徹底失敗！

## 4. 👨‍💻 Bit-Banging (軟體模擬 I2C)
當硬體 I2C 壞掉或不夠用時，韌體工程師可以直接操作 GPIO 腳位的拉高拉低，徒手畫出上述的 START、STOP 與 Data 波形來騙過感測器。在測試階段，這也是一種非常好的 Mocking 練習。
