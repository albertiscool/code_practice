# Day 1: C/C++ 記憶體佈局與指標觀念

## 1. C/C++ 程式的記憶體佈局 (Memory Layout)
當程式執行時，記憶體主要分為四個區段：
- **Data Segment (資料區段)**：存放「有初始值」的全域變數與靜態變數。
- **BSS Segment**：存放「沒有初始值」的全域變數與靜態變數。
- **Heap (堆積區段)**：存放由 `malloc` 或 `new` 動態配置的記憶體。這區的記憶體位址通常是由小往大生長 (向上生長)。
- **Stack (堆疊區段)**：存放函式的區域變數與參數。這區的記憶體位址是由大往小生長 (向下生長)。

*(註：現代 64 位元作業系統如 Windows 有 ASLR 安全機制，會隨機打亂這些區段的起始位置，因此單純比較 Heap 與 Stack 的位址大小無法判斷生長方向，必須在同一個區段內比較變數的先後宣告位址。)*

## 2. 記憶體外洩 (Memory Leak)
- **原因**：使用 `malloc` (C) 或 `new` (C++) 向系統借了 Heap 空間，但在用完後卻忘記呼叫 `free` 或 `delete` 歸還給系統。
- **後果**：隨著程式不斷執行，佔用的可用記憶體會越來越多。如果程式長時間運行（例如伺服器或嵌入式系統），最終會導致記憶體耗盡而崩潰當機。

## 3. 野指標 (Wild Pointer)
- **發生情況**：宣告了一個指標 (例如 `int *p;`) 卻沒有給它初始值，此時該指標內部存放的是一個隨機的垃圾記憶體位址。
- **危險性**：如果直接對野指標進行賦值（如 `*p = 100;`），等於是強迫把值寫入未知的記憶體位置，這會導致程式立刻崩潰 (Segmentation Fault / Access Violation) 或是默默覆寫掉其他重要變數，產生極難除錯的 Bug。

## 4. 指標的正確初始化
為了避免野指標，指標宣告後應採取以下三種作法之一：
1. **指向現有變數**：`int a = 10; int *p = &a;`
2. **動態配置新空間**：`int *p = (int*)malloc(sizeof(int));`
3. **設為空指標等待後續使用**：`int *p = NULL;` 或 `int *p = nullptr;` (C++11推薦寫法)

## 5. 嵌入式系統 (如 STM32) 與 malloc/free
在裸機開發 (Bare-metal) 中強烈不建議頻繁使用 `malloc` / `free`，原因包含：
- **記憶體碎片化 (Fragmentation)**：反覆借還不同大小的空間，會讓 Heap 充滿破碎的空隙，導致最終無法借出連續的大空間。
- **執行時間不可預期**：`malloc` 尋找空閒記憶體的時間會隨 Heap 狀態改變，無法滿足嚴格的即時性 (Real-time) 要求。
- **資源浪費**：STM32 的 SRAM 往往只有幾十 KB，而 `malloc` 管理空間需要額外的記憶體開銷。
- **當機風險**：裸機沒有作業系統保護，一旦記憶體耗盡或分配失敗未處理，整個設備就會卡死。
