# Day 33: Linux 執行緒同步 - 條件變數 (Condition Variables) 與完美防禦

今天我們學習了 Linux 多執行緒開發中最精妙、但也最容易踩坑的同步機制：**條件變數 (Condition Variable)**，並探討了架構師如何透過它打造出完美解耦的「生產者-消費者模型 (Producer-Consumer Model)」。

## 1. 🔑 鎖與條件變數的完美分工
* **互斥鎖 (Mutex)**：負責「空間的安全」。確保我們在讀寫變數時，門是反鎖的，沒有人會衝進來篡改資料（避免 Race Condition）。
* **條件變數 (Cond)**：負責「時間的讓渡」。讓你在條件還沒達成時，主動交出鑰匙去睡覺，把 CPU 讓給別人，彻底消滅了 `while(1)` 瘋狂輪詢 (Polling) 造成的 100% CPU 浪費。

## 2. 🪄 `pthread_cond_wait` 的原子三合一魔法
當消費者呼叫 `wait` 時，Linux 核心會在底層一氣呵成完成三件事：
1. **交出鑰匙 (Unlock)**：把 Mutex 解開，讓生產者有機會進門更新資料。
2. **進入休眠 (Sleep)**：釋放 CPU，直到有人廣播叫醒。
3. **醒來並奪回鑰匙 (Re-lock)**：被叫醒的瞬間，強制重新上鎖，確保後續檢查變數時絕對安全。

## 3. 🛡️ 駭客級防禦：`while` 迴圈與虛假喚醒 (Spurious Wakeup)
* **災難場景**：有時 OS 底層的訊號會意外震醒執行緒，或者在醒來到拿回鑰匙的這微秒間，變數被其他惡意執行緒 (小屁孩) 惡意篡改。若只用 `if` 判斷，會導致帶著錯誤的舊資料引發當機。
* **無敵防禦**：必須將 `wait` 包裝在 `while(電壓 < 3)` 迴圈內。這保證了不管你是怎麼醒來的，拿到鑰匙的第一件事就是「再檢查一次」。若條件其實沒達成，就乖乖放下鑰匙繼續睡！

## 4. 📢 廣播機制：`pthread_cond_broadcast`
* 相比於 `signal` 只能喚醒一個執行緒，`broadcast` 是一鍵廣播。
* 當電壓超標，我們可以用 `broadcast` 瞬間喚醒所有等待的任務（風扇、LED、日誌）。
* **機制陷阱**：雖然全體彈醒，但因為所有任務醒來都必須「重新拿回 Mutex 鑰匙 (Re-lock)」，所以大家依舊會乖乖在門外排隊，依序安全地完成任務，這完美實現了系統模組之間的**極致解耦 (Decoupling)**！
